# Homepage — MCP-First Design Workflow

This page documents how the Packt homepage was recreated in Figma using an
**MCP-first approach** — orchestrating three Figma MCP tools instead of writing
raw Plugin API code by hand.

## Why MCP-first?

When we first built this page, every frame, text node, and section was written
as manual Figma Plugin API JavaScript (hundreds of lines per section). The
MCP-first approach replaces most of that with three tool calls:

| Tool | Role |
|---|---|
| `generate_figma_design` | Captures the live Storybook render as a pixel-perfect Figma frame |
| `use_figma` | Swaps placeholder rectangles for real auto-layout + design-token nodes |
| `get_design_context` | Reads the final Figma frame structurally (layer tree, variables, styles) back into code |

---

## Step 1 — Capture with `generate_figma_design`

With Storybook running on `http://localhost:6006`, point the tool at the
Homepage story URL:

```
generate_figma_design({
  fileKey: "xJmhvrYvw93Y3IqkTPqyJ1",
  url: "http://localhost:6006/?path=/story/pages-homepage--default"
})
```

This renders the full story in a headless browser and imports it into the Figma
file as a raster reference frame — exact spacing, colours, and layout, no
Plugin API code required.

**Trade-off:** the capture is pixel-perfect but contains no Figma auto-layout,
no variable bindings, and no component instances. It is a reference only.

---

## Step 2 — Refine with `use_figma`

Directly above the raster reference, build the real Figma frame using the Plugin
API. For each section, import the matching design-system variable and apply it
instead of a hardcoded value:

```js
// Example: bind the hero background to the brand token
const bgVar = await figma.variables.importVariableByKeyAsync(BRAND_BG_VAR_KEY);
const bgPaint = figma.variables.setBoundVariableForPaint(
  { type: 'SOLID', color: { r: 0, g: 0, b: 0 } },
  'color',
  bgVar
);
heroSection.fills = [bgPaint];
```

Once the `use_figma` frame visually matches the capture (verified with
`await frame.screenshot()`), delete the raster reference:

```js
const captureNode = await figma.getNodeByIdAsync(CAPTURE_FRAME_ID);
captureNode.remove();
```

---

## Step 3 — Read back with `get_design_context`

After the frame is built, `get_design_context` returns the full structural layer
tree along with a screenshot — the same data is used to drive design-to-code
workflows in the other direction:

```
get_design_context({
  fileKey: "xJmhvrYvw93Y3IqkTPqyJ1",
  nodeId: "2225:14901"   // "Packt Homepage — Desktop"
})
```

The response includes node IDs, types, positions, sizes, and variable bindings
for every layer — making it trivial to diff the Figma design against the React
implementation, or to regenerate code from an updated Figma frame.

---

## Step 4 — Type-check with the IDE MCP

Instead of shelling out to `tsc`, the IDE MCP server's `getDiagnostics` surfaces
TypeScript errors inline in the agent context:

```
mcp__ide__getDiagnostics()   // returns [] when clean
```

This was used after fixing prop mismatches in `Homepage.stories.tsx`
(`targetDate` → `endsAt`, `href` → `url`, `facebook` → `github`) to confirm
the file was error-free without a separate terminal step.

## Approach comparison

| | Manual Plugin API | MCP-first |
|---|---|---|
| Lines of JS to write | ~600 | ~50 (refinement only) |
| Visual accuracy | Approximate | Pixel-perfect (raster capture) |
| Design-token bindings | Manual | Manual (same `use_figma` step) |
| Component instances | Yes | Yes |
| Requires live app | No | Yes (`generate_figma_design`) |
| Best for | Offline / no running app | Any app with a local dev server |
