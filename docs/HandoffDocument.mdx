# Handoff Document — Packt Design System

**Version:** 1.0 &nbsp;·&nbsp; **Date:** July 2026 &nbsp;·&nbsp; **Audience:** Designers & Frontend Engineers

This document is the single source of truth for handing off Packt Design System decisions between design and engineering. It covers token architecture, foundations, component inventory, theming rules, and implementation checklists.

---

## 1. Overview

The Packt Design System is a token-based, React + Tailwind component library built on:

| Layer | Technology |
|---|---|
| Token authoring | Tokens Studio (Figma plugin) |
| Token compilation | Style Dictionary 4 |
| Components | React 18+, Radix UI primitives |
| Styling | Tailwind CSS 4 + CSS custom properties |
| Documentation | Storybook 10 |
| Package | `packt-ds` (ESM + CJS) |

> **Rule:** Components must only reference semantic tokens. Never hard-code a primitive hex value or a raw number.

### Token file locations

| File | Purpose |
|---|---|
| `src/tokens/tokens.css` | Primitives + light semantic tokens (always load) |
| `src/tokens/tokens.light.css` | Light mode overrides |
| `src/tokens/tokens.dark.css` | Dark semantic tokens (load when `data-theme="dark"`) |
| `src/tokens/tokens.theme.css` | Theme-scoped shorthand aliases |
| `src/components/tokens.helpers.css` | Component-level helper aliases |

## 3. Theming

### Light / Dark switching

Set `data-theme` on `<html>`:

```ts
// Enable dark mode
document.documentElement.setAttribute('data-theme', 'dark');

// Revert to light
document.documentElement.removeAttribute('data-theme');
```

All semantic tokens automatically resolve to the correct value for the active theme. No class swaps, no JavaScript colour injection.

### Token naming pattern

```
--packt-semantic-colors-{theme}-{category}-{role}
```

| Segment | Examples |
|---|---|
| `theme` | `light`, `dark` |
| `category` | `content`, `background`, `border`, `icon`, `surface`, `overlay` |
| `role` | `primary`, `secondary`, `brand-default`, `error-default`, … |

Shorthand aliases (e.g. `--packt-content-primary`, `--packt-bg-page`) are available via `tokens.helpers.css` for use in component styles.

## 8. Breakpoints

| Name | Token | Width | Notes |
|---|---|---|---|
| Mobile | `--packt-breakpoint-mobile` | 393px | Small phones |
| Tablet | `--packt-breakpoint-tablet` | 768px | Tablets, large phones (landscape) |
| Desktop | `--packt-breakpoint-desktop` | 1024px | Laptops & desktops |
| Large Desktop | `--packt-breakpoint-large-desktop` | 1440px | Wide monitors |

Heading scale steps down at `< 1024px`. Text styles are constant across all breakpoints.

```css
/* Mobile-first */
.component { font-size: 14px; }

@media (min-width: 768px)  { .component { font-size: 16px; } }
@media (min-width: 1024px) { .component { font-size: 18px; } }
```

## 10. Installation & Consumption

```bash
npm install packt-ds
```

### Minimal setup

```ts
// Entry point (e.g. main.ts, _app.tsx, layout.tsx)
import 'packt-ds/tokens';        // primitives + light semantic tokens
import 'packt-ds/tokens/dark';   // dark tokens (activated by data-theme="dark")
```

```tsx
import { Button } from 'packt-ds';

<Button variant="primary" size="md" label="Get started" />
```

### Using tokens in custom CSS

```css
.custom-card {
  background:    var(--packt-bg-surface);
  border:        1px solid var(--packt-border-default);
  border-radius: calc(var(--packt-radius-m) * 1px);
  padding:       calc(var(--packt-space-l) * 1px);
  color:         var(--packt-content-primary);
}
```

> Dimension tokens (`--packt-space-*`, `--packt-radius-*`, `--packt-width-*`) are unitless — wrap in `calc(...* 1px)`.

## 12. Handoff Checklist

### For Designers

- [ ] All new components use semantic tokens — no hard-coded hex or px values
- [ ] Light and dark frames are provided for every new component
- [ ] All interactive states are designed: default, hover, focus, active, disabled, error
- [ ] Spacing uses values from the spacing scale (2px, 4px, 8px, 12px, 16px, 24px, 32px…)
- [ ] Border radius uses a named token value (xs / s / m / l / pill / circle)
- [ ] Typography uses a named text style from the type scale
- [ ] Components are named in Figma to match the component export name in `packt-ds`
- [ ] Any new tokens are added in Token Studio and pushed to the `design-tokens` branch

### For Engineers

- [ ] Component props match the agreed API in the design spec
- [ ] Semantic tokens used throughout — no raw hex or magic numbers
- [ ] Component renders correctly in both light and dark mode
- [ ] All interactive states implemented (default, hover, focus, active, disabled, error)
- [ ] Keyboard navigation works (Tab, Enter, Escape, Arrow keys where applicable)
- [ ] ARIA attributes present for interactive elements
- [ ] Story written covering: Default, all variants, all sizes, disabled, dark mode
- [ ] `export` added to `src/index.ts`
- [ ] No peer-dependency changes without team sign-off

## 13. Token Pipeline

Token changes flow automatically when pushed to the `design-tokens` branch:

```
Token Studio (Figma)
        ↓  push
design-tokens branch
        ↓  GitHub Action
Style Dictionary compile
        ↓
src/tokens/tokens.css (committed)
        ↓  merge to main
packt-ds package
```

Manual regeneration (local only):

```bash
npm run tokens
```

See `docs/TOKEN-PIPELINE.md` and `docs/EXPORT_GUIDE.md` for the full workflow.

## 14. Accessibility Standards

All components target **WCAG 2.1 AA**:

| Requirement | Implementation |
|---|---|
| Colour contrast | Semantic colour pairings verified ≥ 4.5:1 (text) / 3:1 (UI) |
| Keyboard access | All interactive elements reachable and operable via keyboard |
| Focus indicators | `focus-visible` outline using `--packt-border-brand` at 2px |
| Screen reader support | Radix primitives provide ARIA roles; custom components use semantic HTML |
| Touch targets | Minimum 44 × 44px for all interactive elements |

Run the a11y addon checks in Storybook (the shield icon on the toolbar) to audit any story.
